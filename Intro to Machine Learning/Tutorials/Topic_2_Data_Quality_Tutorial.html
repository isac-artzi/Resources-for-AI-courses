Topic 2 · Data Pipelines & Data Quality

Topic 2 is the bridge between "I have a CSV" and "I have a model." You will profile a messy raw dataset, write a small pandera schema, decide what to do about each kind of problem you find, apply pandas cleaning functions in small named steps, and persist the cleaned result as a versioned CSV ready for downstream use.

Scope at the introductory level: profiling (row count, null %, distinct count, range), a small pandera schema (dtype + nullability + a simple range check), three or four named cleaning functions, and the versioned-CSV persist habit. Out of scope: cross-column validation, regex patterns inside schemas, and outlier-detection statistics.

1. ETL vs ELT in one minute

Two patterns describe how raw data ends up in an analytics layer. Both are introduced in this course because you will hear them in interviews; in your projects you only build the simpler "ETL-with-pandas" version.

PatternOrderWhere the work happensWhen it fits
ETLExtract → Transform → LoadTransformation in Python / pandas before the warehouse stores itSmall files, well-known schema, you control the destination.
ELTExtract → Load → TransformRaw lands in the warehouse; transforms run as SQL inside the warehouse laterCloud warehouses with elastic compute. Dominant in modern stacks.

2. Why profile first?

Every cleaning decision starts with looking. If you do not know how many rows the file has, how many nulls each column has, or what its distinct values look like, you cannot write a sensible cleaning function. The introductory profile has just five quantities per column:

WhatWhy you compute itpandas one-liner
row countSanity check: does this look like the file we expected?len(df)
null fraction per columnFind missingness early; decide impute vs drop vs flag.df.isna().mean()
distinct count per columnTells you which columns are categorical vs free-text.df.nunique()
min / max / mean (numeric)Catches impossible values: negative ages, scores above 1.df.describe()
dtypesConfirms pandas inferred the right type for each column.df.dtypes

4. Outlier detection at the introductory level

5. pandera — your data contract

A pandera schema is a small Python object that says "every row should look like this." When the cleaned data is passed through the schema, any violation raises an error — so silent corruption becomes loud and visible at the moment it appears. 

import pandera as pa
from pandera import Column, DataFrameSchema, Check

schema = DataFrameSchema({
    "age":     Column(int,   Check(lambda s: s.between(0, 120))),
    "country": Column(str,   nullable=False),
    "score":   Column(float, nullable=False,
                      checks=[Check.ge(0), Check.le(1)]),
})

schema.validate(df)   # raises if any row breaks the contract

7. The cleaning pipeline as a small Python module

Every cleaning step in this course is a named, single-responsibility function with a docstring and at least a smoke test. Tiny functions make the pipeline easy to read, easy to test, and easy to swap when the rules change. 

def clean_amount(s: pd.Series) -> pd.Series:
    """Replace negative amounts with pd.NA; preserve existing NAs."""
    return s.where(s >= 0, other=pd.NA)

def clip_age(s: pd.Series) -> pd.Series:
    """Clip ages to [0, 120]; values outside are clipped, not dropped."""
    return s.clip(lower=0, upper=120)

def drop_obvious_duplicates(df: pd.DataFrame) -> pd.DataFrame:
    """Drop duplicate rows that share the same (user_id, ts)."""
    return df.drop_duplicates(subset=["user_id", "ts"], keep="last")

def coerce_date(s: pd.Series) -> pd.Series:
    """Parse strings to datetime; unparseable values become NaT."""
    return pd.to_datetime(s, errors="coerce")

Compose them in a single clean() function and call that once from your build script. Each step is independently testable; the assembly is just a sequence of calls.

8. Provenance — keep receipts

Provenance is the record of where the data came from and what was done to it. At the introductory level, you keep three pieces of information next to every cleaned dataset:

  1. The source URL (or filesystem path) of the raw file.
  2. The retrieval date.
  3. The list of cleaning steps that produced the cleaned file.

Pin them in a small data/README.md. Future you, your teammates, and your stakeholders will thank you when something surprising happens downstream and they need to track the value back to its origin.

9. Persisting the cleaned DataFrame

Once the cleaning functions have run, write the cleaned DataFrame to a versioned CSV under data/derived/. The filename convention makes the artifact reproducible and the date visible on the file system without opening the file. 

from datetime import date
fname = f"data/derived/cleaned_orders_{date.today():%Y%m%d}.csv"
cleaned.to_csv(fname, index=False)

Why index=False? Without it, pandas writes its row index as the first column of the CSV. That column then re-appears (named Unnamed: 0) every time you read the file back. Just say no.

10. Ethics: silent failures vs loud failures

The strongest argument for programmatic data validation is not speed — it is auditability. A pandera schema makes the team's assumptions about the data explicit, in code, and re-checked on every commit. When a future version of the upstream system changes the semantics of a column (cents become dollars; "M/F" become "1/2"), the schema catches the change before a model trained on the new semantics is shown to a real person.

Silent failures are the dangerous ones. Catching them takes one afternoon of writing a schema and pays you back forever.
