Topic 1 · SQL Basics & Reading Relational Data into pandas

Topic 1 is your introduction to handling real data. By the end of this tutorial you will understand what a relational database is, how to load and query it with basic SQL, how to read its tables into pandas, and how to combine and clean those tables — entirely in pandas — using the build pattern.

Scope at the introductory level: CREATE TABLE · INSERT · single-table SELECT (with WHERE / ORDER BY / LIMIT / simple GROUP BY) · UPDATE · DELETE. JOINs, HAVING, subqueries, window functions, and CTEs are out of scope — all multi-table work happens in pandas.

1. What is a relational table?

A relational table is a grid of values — rows are records and columns are attributes. Every column has a fixed type (integer, text, etc.) and every row in a well-designed table has a primary key that uniquely identifies it. Several tables can be related to each other through shared column values (a customer_id in the invoices table refers to the customer_id in the customers table).

PK / FK: A primary key uniquely identifies a row inside one table. A foreign key is a column whose value points to a primary key in another table — it is how relational databases "link" tables together. We follow those links in pandas, not in SQL.

3. SELECT — the read operation in detail

Almost every interaction with a database starts with a SELECT. SELECT has a simple anatomy at the introductory level: 

SELECT <columns>
FROM   <one table>
WHERE  <row filter>
GROUP BY <column>            -- optional, for COUNT/SUM per group
ORDER BY <column> [ASC|DESC] -- optional
LIMIT  <n>                  -- optional

4. WHERE operators you need at the introductory level

WHERE filters rows before any aggregation. The operators below cover almost everything you will need in this course.

OperatorExampleMeaning
=, !=country = 'US'Exact match (and its negation).
<, <=, >, >=total > 100Numeric comparison.
LIKEname LIKE 'A%'Pattern match. % stands for any string; _ for a single char.

IS NULLcountry IS NULLTest for missing values. Do not use = for NULL — it always returns false.

Watch out: SQL strings are enclosed in single quotes 'US', not double quotes. And NULL is not equal to anything — even to itself — so x = NULL never matches. Use IS NULL instead.

6. Reading a SQLite table into pandas

Once data is in SQLite, the standard way to work with it is: read each table into its own pandas DataFrame and do everything else in pandas. 

import sqlite3, pandas as pd

con = sqlite3.connect("db/chinook.sqlite")

# read each table into its own DataFrame
customers = pd.read_sql("SELECT * FROM customers", con)
invoices  = pd.read_sql("SELECT * FROM invoices",  con)

con.close()

print(customers.shape)   # (rows, columns)
print(customers.head())  # first 5 rows
print(customers.dtypes)  # column types as pandas inferred them

Three things to notice:

  • pd.read_sql accepts any SQL string. We keep ours simple.
  • Each call touches one table. There is no JOIN.
  • It returns a regular pandas DataFrame, so every method you have learned in pandas works directly.

11. Stewardship: every filter changes the population

Every WHERE clause (or pandas boolean filter) silently changes the population the report represents. A report that says "average revenue per customer is $120" is misleading if the underlying query quietly excluded all customers in the smallest country. At the introductory level, stewardship starts with one habit: name the rows you kept, name the rows you dropped, and say why.

Compose a one-paragraph "rows-and-reasons" note for every analytical page in your Streamlit app. Future you, your team, and your stakeholders all benefit when the population is no longer hidden.
