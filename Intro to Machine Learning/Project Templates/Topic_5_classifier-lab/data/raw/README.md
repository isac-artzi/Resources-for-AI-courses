# Raw data — `customers.csv`

A **synthetic** customer-churn dataset. It is generated deterministically
(fixed random seed) by [`../../db/generate_raw.py`](../../db/generate_raw.py),
so re-running that script always produces this exact file and the tests stay
reproducible. You normally do **not** need to regenerate it — it is committed.

## Why synthetic (and how to use a real dataset)

A real churn dataset would work too, but a synthetic one lets us control the
signal-to-noise ratio so the three classifiers are all decent yet clearly
different. The label comes from a hidden logistic rule plus a little noise
(so no model is perfect). To use real data, replace `generate()` in
`db/generate_raw.py` with a loader that returns the same feature columns plus a
0/1 `churned` column.

## Provenance

- **Source:** Synthetic customer-churn data generated by `db/generate_raw.py`.
- **License:** Public domain (synthetic — no real customers).
- A copy of this provenance is written into a `provenance` table when the
  SQLite database is built.

## Table dictionary — `customers`

| Column            | Type | Notes                                          |
|-------------------|------|------------------------------------------------|
| `tenure_months`   | int  | Months as a customer (1–72).                   |
| `monthly_charges` | float| Monthly bill in dollars (15–200).              |
| `support_calls`   | int  | Number of support calls made.                  |
| `age`             | int  | Customer age (18–80).                          |
| `num_services`    | int  | Number of add-on services (1–6).               |
| `satisfaction`    | int  | Survey score (1–10).                           |
| `churned`         | int  | **Target.** 1 = left the service, 0 = stayed.  |

2000 rows. Churn is more likely for customers with many support calls, low
satisfaction, high charges, and short tenure.
