# Raw data — `signups.csv`

A **synthetic**, deliberately messy customer-signup dataset. It is generated
deterministically (fixed random seed) by [`../../db/generate_raw.py`](../../db/generate_raw.py),
so re-running that script always produces this exact file and the tests stay
reproducible. You normally do **not** need to regenerate it — it is committed.

## Provenance

- **Source:** Synthetic data generated by `db/generate_raw.py`.
- **License:** Public domain (synthetic — no real people).
- A copy of this provenance is written into a `provenance` table when the
  SQLite database is built.

## Table dictionary — `signups`

| Column        | Type   | Notes                                                        |
|---------------|--------|--------------------------------------------------------------|
| `signup_id`   | int    | Unique row id (1..N). Duplicated rows reuse an existing id.  |
| `name`        | text   | `User <n>`.                                                  |
| `age`         | float  | Mostly 18–70; ~5% missing; a few impossible `999` values.    |
| `country`     | text   | Inconsistent casing/whitespace (`USA`, `usa `, ` Canada`…).  |
| `signup_date` | text   | `YYYY-MM-DD` string (as it would arrive from a CSV export).  |
| `income`      | float  | ~N(70000, 25000); ~5% missing; a few negative outliers.      |
| `plan`        | text   | Inconsistent casing (`free`, `Free`, `pro`, `PRO`, …).       |

## Injected quality problems (on purpose)

1. **Missing values** — ~5% of `age` and `income` blanked out.
2. **Outliers** — a few `age = 999` and `income = -5000` values.
3. **Inconsistent text** — mixed casing and whitespace in `country` and `plan`.
4. **Duplicates** — 15 random rows copied and appended (so 400 → 415 rows).

The whole point of Topic 2 is to *measure* these problems, *validate* against a
schema that rejects them, and *clean* them away.
