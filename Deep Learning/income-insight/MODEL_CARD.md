# Model Card — Income-Insight

## Model details
- **Type:** feed-forward neural network (multi-layer perceptron) for binary
  classification, `p = σ(W₂·ReLU(W₁x + b₁) + b₂)`, trained with Adam + binary
  cross-entropy (PyTorch).
- **Preprocessing:** an sklearn `ColumnTransformer` — `StandardScaler` on numeric
  features and `OneHotEncoder` on categorical features — fit on the training split
  and serialized alongside the network.
- **Training:** per-run hyperparameters (hidden width, learning rate, batch size,
  epochs) supplied by the API request.
- **Owner:** your team (add your names).

## Intended use
- **Primary:** an interactive teaching demo of a tabular classifier and the
  train → evaluate → serve → audit loop across three clouds.
- **Out of scope:** any real income, hiring, lending, or eligibility decision.
  The model is trained on **synthetic** data with a known generating rule.

## Data
- Synthetic records shaped like the UCI *Adult Income* schema: numeric
  (`age`, `education_num`, `hours_per_week`, `capital_gain`) and categorical
  (`workclass`, `marital_status`, `occupation`) features with a binary `income`
  target (`<=50K` / `>50K`).
- Generated by `shared/data.generate_tabular` from a fixed logistic rule with a
  small amount of label noise; stored in the Supabase `datasets` table.
- **No real personal data** is used or stored — every row is fabricated.

## Metrics
- Reported on a held-out 20% test split: **accuracy, precision, recall, F1,
  ROC-AUC**.
- Every run's metrics are persisted to the Supabase `runs` table (single source
  of truth — not recomputed in the UI).

## Fairness & monitoring
- The `/audit` endpoint groups a run's logged predictions by a categorical
  feature and reports the positive-prediction rate per group, so disparate
  outcomes are visible. On synthetic data this is illustrative only.
- On real data you would treat `marital_status` and similar attributes as
  sensitive, check for disparate impact, and never deploy without a documented
  fairness review.

## Limitations
- A shallow MLP on a handful of features; it will miss interactions a richer
  model or better features could capture.
- Sensitive to learning rate and hidden width: too large an lr can destabilize
  training; too few epochs under-fits.
- Returns a probability but no calibrated uncertainty guarantee.

## Ethical considerations
- A non-technical client must be told the model's limits plainly
  (honesty/stewardship): it is a demonstration on synthetic data, not a validated
  predictor of anyone's income. Disclose the assumptions, the held-out metrics,
  and the fairness caveats; do not overstate confidence.

## What we log (and don't)
- We log run hyperparameters, metrics, the fitted model artifact, and each
  prediction `(run_id, features, proba, label)` for auditability.
- The dataset is synthetic by construction, so no real PII is ever logged.
