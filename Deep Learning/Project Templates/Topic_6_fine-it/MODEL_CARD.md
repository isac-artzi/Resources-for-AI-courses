# Model Card — Fine-It

## Model details
- **Type:** a from-scratch **causal character Transformer** (decoder-style) —
  token embedding + sinusoidal positional encoding → stacked pre-norm blocks
  (causal-masked multi-head self-attention + feed-forward, each with a residual
  connection and layer norm). The shared trunk feeds **two heads**: an **LM head**
  for next-character prediction and a **classification head** (mean-pool → linear).
  Trained with Adam + cross-entropy (PyTorch); no `nn.Transformer`.
- **Size:** `d_model=48`, `n_heads=3`, `n_layers=2`, `d_ff=96` by default.
- **Two phases (transfer learning):**
  1. **Pretrain** the trunk + LM head with self-supervised next-character
     prediction on an unlabelled corpus.
  2. **Fine-tune** a classifier head twice on a small labelled set — once
     **warm-started** from the pretrained trunk, once **from scratch** — so the
     transfer gap is visible.
- **Explainability / interactivity:** the LM head **generates** text by
  temperature sampling, and the Fine-tune tab shows the **pretrained-vs-scratch
  accuracy bar** that makes the value of pretraining concrete.
- **Owner:** your team (add your names).

## Intended use
- **Primary:** an interactive teaching demo of self-supervised pretraining,
  fine-tuning, and the transfer gap, across three clouds.
- **Out of scope:** any real-world language task. The model is trained on
  **synthetic** character sequences with a known generating grammar.

## Data
- Synthetic character strings (alphabet `abcd`, default length 20) drawn from
  `N_CLASSES` **dialects**. Each dialect is a first-order Markov chain whose
  transition matrix is `STAY * cyclic-shift + (1-STAY) * uniform`. The matrices
  are **doubly stochastic**, so every dialect has the *same* uniform unigram
  (character-frequency) distribution and differs ONLY in its transitions — a
  bag-of-characters view cannot tell them apart.
- Two views share the alphabet: an **unlabelled** corpus for LM pretraining and a
  small **labelled** set (string → dialect) for fine-tuning.
- Generated by `shared/data.py` deterministically from a seed; only the
  generation **parameters** (`n_rows`, `seq_len`, `seed`) are stored in the
  Supabase `datasets` table — never the strings.
- **No real text or personal data** are used or stored.

## Metrics
- **Pretraining:** held-out next-character cross-entropy (`val_loss`).
- **Fine-tuning:** held-out **accuracy** and **macro-F1** (computed manually, no
  sklearn dependency in the API) for BOTH the warm-started and the from-scratch
  model, plus the from-scratch **scratch_accuracy** baseline.
- Every run's metrics are persisted to the Supabase `runs` table (single source
  of truth — not recomputed in the UI). A `run_type` column marks pretrain vs
  finetune.

## Privacy — what we log (and don't)
- For every served classification we store the string's **sha256 hash + length +
  predicted class + confidence** in `sequence_metadata` — **never the raw
  characters** (the "store hashes, not raw payloads" invariant).
- Fitted models (base64-pickled) live in `run_artifacts`, which is **not**
  anon-readable; only run metrics are exposed to the UI's anon key.

## Limitations
- A compact Transformer on a toy dialect task; it will not generalize to natural
  language, and the generated text is only as rich as the synthetic grammar.
- The transfer gap depends on the labelled set staying **small**: with enough
  labelled data the from-scratch baseline catches up (which is itself the point —
  pretraining buys data efficiency, not a permanent edge).
- Sensitive to learning rate and epochs: too little pretraining leaves the trunk
  with weak features and the gap shrinks.

## Ethical considerations
- A non-technical client must be told the model's limits plainly
  (honesty/stewardship): it is a demonstration on synthetic data, not a validated
  language system. The pretrained-vs-scratch bar builds intuition for *why*
  pretraining helps, but disclose the held-out metrics and the small-labelled-set
  caveat alongside it.
